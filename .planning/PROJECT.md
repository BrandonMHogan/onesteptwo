# OneStepTwo

## Overview

OneStepTwo is an offline-first mobile app (Android + iOS) for potty training tracking. Families create a shared child profile; multiple caregivers log potty events from any device and receive push notifications when any caregiver logs a new event. Events are written locally to SQLDelight immediately and synced to a Go backend on Railway when connectivity is available. The architecture is privacy-first by design: the only child data stored is nickname plus birth month and year, with parent and caregiver PII living exclusively in Clerk.

## Goals

- Enable multi-caregiver families to coordinate potty training across different locations and caregivers
- Log potty events quickly (one-tap) with optional detail added later
- Deliver timely push notifications to all opted-in family members when a new event is logged
- Operate fully offline — no connectivity required to log an event
- Minimise child PII by design (nickname + birth month/year only) to satisfy COPPA, GDPR, PIPEDA, and POPIA
- Support family onboarding via Clerk invitation flow — no custom invitation backend required
- Provide a stable, versioned API that supports old app versions in the field during staged rollouts

## Non-Goals (v1)

- Real-time sync (no WebSockets or SSE — pull-to-refresh only)
- Web dashboard for parents (marketing site only in v1; dashboard deferred to v2)
- Photo or media storage (child media has high privacy risk; deferred to v2)
- Background sync when app is closed (deferred to v2)
- Data portability export endpoint (GDPR Article 20; deferred to v2)
- MFA on free Clerk tier (Pro plan required; not launch-blocking for a family tracking app)
- Rate limiting in v1 (to be added as Go middleware before scaling)
- Automatic data expiry (user-initiated deletion is the retention mechanism)
- Verified parental consent (VPC) beyond self-attestation checkbox (verify with legal counsel per market)
- EU data residency (Railway EU region to be evaluated when EU user base scales)

## Success Metric (v1)

Both Android + iOS apps published live on Play Store and App Store, full family creation through push notifications working end-to-end.

## Stack

- **Android**: Jetpack Compose (native), applicationId: com.onesteptwo.android, minSdk 29, compileSdk/targetSdk 36
- **iOS**: SwiftUI (native), bundle ID: com.onesteptwo.ios, deployment target 17.0
- **Shared logic**: Kotlin Multiplatform, module :shared, Kotlin 2.0.21 (pinned), Java toolchain 21
- **Swift interop**: SKIE (Touchlab) — no KMP-NativeCoroutines, no Swift Export; use concrete typed classes at the KMP/Swift boundary
- **On-device storage**: SQLDelight with .sqm migrations starting at 1.sqm on day one
- **HTTP client**: Ktor Client in commonMain of :shared
- **Backend**: Go single binary, module path github.com/BrandonMHogan/onesteptwo/backend
- **API contract**: OpenAPI spec at /api/openapi.yaml; stubs generated by oapi-codegen (make generate), generated.go committed
- **Deployment**: Railway always-on (no sleep); two environments — staging (local Go run) and production (Railway)
- **Database**: PostgreSQL on Railway, two separate instances (staging + production)
- **Migrations**: goose with `-- +goose Up / Down` markers, run via `go run cmd/migrate/main.go`, staging-first
- **Backups**: daily pg_dump → Cloudflare R2 via Railway cron (configured day one)
- **Auth**: Clerk (headless); clerk-android-api on Android, ClerkKit on iOS, clerk-sdk-go v2 on Go; Clerk Organizations = families
- **Push**: FCM unified delivery layer; one FCM call delivers to Android + iOS via APNs bridge; two Firebase projects (staging, production)
- **Web**: SvelteKit on Cloudflare Pages — v1 scope is marketing landing page only
- **API errors**: RFC 7807 Problem Details (application/problem+json)
- **API versioning**: /v1/ prefix; breaking changes introduce /v2/

## Constraints

**Regulatory (hard gates):**
- A consent_events row must exist in the database before any children row is created — enforced at DB and application layer; this is a legal gate, not a UX preference
- Legal erasure (GDPR/COPPA) must use real DELETE statements — no soft-delete, no orphaned rows
- No IP addresses stored in consent_events

**Technical (hard constraints):**
- Android minSdk 29, iOS deployment target 17.0, Java toolchain 21, Kotlin 2.0.21 (pinned — upgrade deliberately with SKIE together)
- SKIE only — do not mix with KMP-NativeCoroutines; the two have conflicting cancellation behavior
- Do not expose generic types at the KMP/Swift boundary; use concrete typed repository classes
- Use sealed classes for error states at the KMP boundary (Kotlin.Result<T> does not bridge to Swift Result<T>)
- Clerk SDK v2 only (github.com/clerk/clerk-sdk-go/v2, v2.7.0+) — v1 is EOL April 2025
- CLERK_AUTHORIZED_PARTY must be set explicitly in production — azp is not validated by default
- Always use "org:admin" and "org:caregiver" role strings with the org: prefix — bare role strings cause silent failures
- FCM service account JSON must never be committed; stored as Railway env var FCM_SERVICE_ACCOUNT_JSON
- SQLDelight .sqm migrations start at 1.sqm on day one with no numbering gaps
- No direct APNs calls — FCM is the unified delivery layer

**Operational:**
- Postgres backup (pg_dump → R2) configured on day one — Railway has no built-in PITR
- Two separate Firebase projects for staging and production — no shared state between environments
- Production env vars in Railway dashboard only — never in the repo
- JVM memory flags must be set in Railway env (relevant if JVM tooling is added to the Railway environment)
