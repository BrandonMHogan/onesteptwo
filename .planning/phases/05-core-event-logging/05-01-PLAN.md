Implement Stage 1 of Phase 5 (Core Event Logging) for the Android app: theme foundation, the SQLDelight data layer (driver, queries, repositories), the onboarding wizard, the 4-tab navigation shell, and a fully working Home tab (one-tap log -> toast -> chip-pick / add-details -> bottom sheet -> save, plus child switcher). That mockup is a visual/interaction reference only — its 6-value event-type chip set is superseded by the 5-value enum locked in `05-CONTEXT.md` D-04 (`pee, poo, both, accident, tried`).

Out of scope for this plan (see `05-02-PLAN.md`): History tab heatmap + day-detail + event delete, Settings tab child management + notification preferences + account actions, `PATCH /v1/children/{id}`.

<decisions>
## Architecture decisions for this plan

- **No DI framework.** A small `AppContainer` plain class, constructed once in `ClerkApp.kt`, holds the `SqlDriver`/`OneStepTwoDatabase` and the three repositories. Screens obtain ViewModels via `viewModel(factory = remember { XViewModelFactory(app.container) })`. Matches the app's existing zero-framework style (no Koin/Hilt/Dagger anywhere today).
- **Repository per table** in `shared/src/commonMain/kotlin/com/onesteptwo/data/`: `ChildrenRepository`, `PottyEventsRepository`, `ConsentEventsRepository`. Reads expose `Flow<...>` via SQLDelight `coroutines-extensions` (`.asFlow().mapToList()`); writes are suspend functions wrapping `.sq` statements.
- **Single nested NavHost** for the 4-tab shell — routes `home`, `history`, `history/day/{date}`, `progress`, `settings` inside one `Scaffold` + `NavigationBar`, not one NavHost per tab.
- **Onboarding is one route with an internal step enum** (FamilyName -> ChildDetails -> Consent -> Success), not a nested nav graph — steps share an in-progress draft.
- **Active child selection** lives in `ChildSelectionViewModel`, scoped above the tab NavHost (constructed once in `AppNavigation`), shared by Home/History/Progress.
- `GET /v1/children` is pulled forward from Stage 2 because the post-auth routing decision (wizard vs. main shell) has no other data source for a returning/invited caregiver with zero local rows.
</decisions>
