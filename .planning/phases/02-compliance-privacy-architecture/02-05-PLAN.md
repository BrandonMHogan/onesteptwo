<objective>
Close VERIFICATION.md Gap 1 (REQ-NF-001 violation, code-review CR-02, BLOCKER). The server currently registers routes with `api.HandlerFromMux(srv, mux)`, which uses oapi-codegen's generated default `ErrorHandlerFunc` (`generated.go` lines 259-263). That default calls `http.Error()`, returning `Content-Type: text/plain` and the raw Go error string for path-parameter binding failures — for example a non-UUID `{id}` in `DELETE /v1/children/{id}`. REQ-NF-001 requires every API error to be RFC 7807 `application/problem+json`. This plan adds a custom `ProblemErrorHandler` that routes binding failures through the existing `WriteProblem` helper, and wires it via `api.HandlerWithOptions`.

Purpose: REQ-NF-001 (all API errors must be RFC 7807 Problem Details) — this is live code, not a deferred concern, so it must be fixed in Phase 2. REQ-013 deletion endpoints are the endpoints whose path params currently leak text/plain.
Output: A tested `ProblemErrorHandler` in the api package and a one-line wiring change in main.go.
</objective>

| T-2-13 | Tampering | inconsistent error content-type lets clients/proxies mis-handle errors | mitigate | All binding failures funnel through WriteProblem which sets Content-Type application/problem+json and mirrors the status |
| T-2-SC | Tampering | supply-chain (package installs) | accept | No new packages introduced — uses stdlib errors + existing generated types |
