<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → Clerk org API | Org activation and invitations handled by Clerk; app holds no membership table |
| admin → invitation | Only org:admin may invite; role enforced client-side (UX) and server-side (03-01) |
| caregiver session → API | Caregiver JWT carries org:caregiver; server rejects admin-only ops |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-05 | Elevation of Privilege | invite action visible to caregivers | mitigate | Invite route gated on `org:admin`; server also enforces HasRole (REQ-016, 03-01) |
| T-3-13 | Tampering | invitation to a non-existent role | mitigate | `org:caregiver` role pre-created in Dashboard (Task 1, Pitfall 5); exact role key used |
| T-3-14 | Spoofing | stale org context after switching | mitigate | `setActive` reissues the JWT with the new org_id; subsequent calls carry it |
| T-3-15 | Information Disclosure | invitation error leaking member PII | accept | UI shows fixed strings ("already in your family"); no raw Clerk error surfaced |
</threat_model>
