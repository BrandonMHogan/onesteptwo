<objective>
Make the Android app authenticate. Initialize the Clerk Android SDK in an `Application` subclass, host an auth-gated navigation graph in `MainActivity`, and build the Sign In and Sign Up screens (email/password via the Clerk headless SDK) plus a minimal post-auth stub. After a successful sign-in, activate the user's single org and land on the stub; capture the native-app `azp` claim from a real JWT and set `CLERK_AUTHORIZED_PARTY` so the Go middleware (03-01) validates it (REQ-026).

Purpose: This is the first real end-to-end identity path — a user signs up/in on Android, the Clerk session token flows through the shared HttpClient (03-02) to the Go API (03-01). It also resolves the open `azp` value question that the server middleware depends on.

Output: Clerk init plumbing; auth-gated `AppNavigation`; `SignInScreen` + `SignUpScreen` per UI-SPEC; post-auth stub; a captured/verified `CLERK_AUTHORIZED_PARTY`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → Clerk Android SDK | Credentials handled entirely by Clerk; app never stores passwords |
| Android app → Go API | Bearer JWT; azp must match CLERK_AUTHORIZED_PARTY server-side |
| BuildConfig → APK | Publishable key embedded (public by design); secret key never on device |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-02 | Elevation of Privilege | azp validation dependency | mitigate | Capture real native azp; set CLERK_AUTHORIZED_PARTY in Railway (REQ-026); verified in Task 3 |
| T-3-10 | Information Disclosure | temporary JWT logging during azp capture | mitigate | Token logging is debug-only and removed before task completion (Task 3 step 7) |
| T-3-11 | Spoofing | uninitialized Clerk SDK returning null token | mitigate | Clerk.initialize in Application subclass (RESEARCH Pitfall 7) |
| T-3-12 | Information Disclosure | publishable vs secret key confusion | accept | Only the publishable key (pk_) is embedded; it is public by Clerk design; secret key stays server-side |
</threat_model>
