<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External dashboards (Railway/Cloudflare/Clerk/Firebase) → repo | Provisioning produces credentials that must live in env/secret stores, never in git |
| Internet → Railway service → Postgres (private network) | Public traffic reaches /healthz; Postgres is reachable only over Railway's private network |
| Staging vs production isolation | Separate services, Postgres instances, Clerk apps, and Firebase projects must not share state |

## STRIDE Threat Register

| Threat ID | Category | Severity | Component | Disposition | Mitigation Plan |
|-----------|----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | high | Secrets (DATABASE_URL, R2 keys, Clerk sk_*, FCM service account, production google-services.json) | mitigate | All secrets live only in Railway env / GitHub Secrets; `.gitignore` excludes release google-services.json; blocking human-action checkpoints confirm nothing secret is committed before resume |
| T-04-02 | Information Disclosure | high | Production google-services.json committed by mistake | mitigate | Production config stored as a GitHub Secret only; `git check-ignore` on `androidApp/src/release/google-services.json` verified in acceptance criteria; only the non-sensitive staging file is committed |
| T-04-03 | Spoofing / Tampering | high | Postgres public exposure | mitigate | Postgres reachable only via Railway private network (PROJECT.md locked); services connect through injected DATABASE_URL, no public DB port |
| T-04-04 | Tampering | medium | Staging/production environment cross-contamination | mitigate | Separate Railway services, separate Postgres instances, separate Clerk apps, and two Firebase projects (PROJECT.md); D-04 ties each service to its own branch |
| T-04-05 | Information Disclosure | low | /healthz on public Railway URL | accept | Endpoint returns 200 empty body (REQ-001) — no internal info; acceptable per T-01-01 |
</threat_model>
